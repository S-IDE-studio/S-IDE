# ARCHITECTURE.md — S-IDE 憲法

本ドキュメントはS-IDEプロジェクトの**根本思想**を定義する。
実装詳細は記述しない。すべての判断はここに記された原理から導出せよ。

---

## 1. Mission

S-IDEは**AIエージェント駆動開発の確実性を極限まで引き上げるインフラストラクチャ**である。

確率的なAIモデルに対し、決定論的な境界・観測・介入を提供することで、
ゼロトラストなタスク委譲と人間の生産性の飛躍的向上を実現する。

S-IDEは**IDEではない**。AIエージェントのオーケストレーションと統合管理を行うための制御基盤である。

---

## 2. Core Philosophy（基本理念）

Missionを達成するためのアプローチは、以下の**2つの柱**に集約される。
すべてのアーキテクチャ判断は、この2つの理念のいずれか、または両方に奉仕しなければならない。

### P-1: Deterministic Orchestration（決定論的オーケストレーションと絶対的介入権）

AIモデル自体は確率的であるため、S-IDEは厳格な境界と予測可能なI/Oを持つ**「決定論的な環境」**として振る舞う。

絶対的な可観測性と介入権を持ち、閾値を超えた場合に即座にプロセスを遮断・軌道修正することで、**ゼロトラストなタスク委譲**を実現する。

これは以下を意味する:
- S-IDE自体の振る舞いは、同じ入力に対して同じ出力を返す
- すべてのエージェント実行は、外部から観測可能で、いつでも遮断可能である
- エージェント間のすべての通信は、記録可能・傍受可能・遮断可能な単一の契約レイヤーを通る
- UIの障害がエージェントの監視・制御を妨げることは許されない

### P-2: Integrated AI Agent Management（AIエージェントの統合管理）

AIを単なるプロセスとして扱わず、**「固有の能力と役割を持つソフトウェア資産」**として体系的に管理する。

MCPツール、プロンプト、設定、依存関係を明確に定義し、バージョン管理・再現可能な構造を持つ。

これは以下を意味する:
- 各エージェントはIDで識別され、能力・設定・スキルが構造化されたデータとして管理される
- エージェントへの情報提供は、有限なコンテキストウィンドウを考慮し、最小・最適な形で行われる
- 環境の再現性が保証され、任意の環境で同じエージェント構成をデプロイできる
- エージェントのコスト・パフォーマンスは定量的に計測・比較可能である

### 2.1. 理念の優先順位（Hierarchy）

> **P-1は、P-2よりも常に優先される。**

安全と制御（P-1）は、効率と管理（P-2）に対して**絶対的な優位性**を持つ。

この優先順位は、Invariants同士がコンフリクトした場合の判断基準となる。
P-1に奉仕するルール（INV-1, INV-2, INV-3, INV-6）は、P-2に奉仕するルール（INV-4, INV-5）よりも優先される。

**具体例:**
- INV-5（Context Economy）を追求して情報を削りすぎた結果、INV-6（Observable & Interruptible）が損なわれ、エージェントの暴走を検知できなくなる場合 → **INV-6が勝つ。** 可観測性を犠牲にしてはならない。冗長でもモニタリングデータは保持せよ。
- INV-4（Config-as-Code）の設定の柔軟性を高めた結果、INV-2（Deterministic Boundary）の決定的な振る舞いが損なわれる場合 → **INV-2が勝つ。** 設定のバリエーションを制限してでも、決定論的な出力を保証せよ。

**判定フロー:**
```
Invariant同士がコンフリクトした場合:

1. 両方がP-1に奉仕する → トレードオフを明文化し、人間に判断を仰げ
2. 一方がP-1、他方がP-2 → P-1側が無条件で勝つ
3. 両方がP-2に奉仕する → Evaluation Framework (§4) のスコアで判断
```

## 3. Invariants（不変条件）

以下の6原則は、§2の基本理念を具現化するための**物理的な実装ルール**である。
各ルールは独立して存在するのではなく、理念を実現する手段として論理的に接続されている。

エージェントはルールを暗記するのではなく、**どの理念を守るためのルールか**を理解して判断に用いよ。
ルール同士がコンフリクトした場合は、§2.1 の優先順位に従え。

### INV-1: Headless-First（頭脳とUIの完全分離）

> **すべてのビジネスロジックはUIなしで動作しなければならない。**

**奉仕する理念:** P-1（決定論的オーケストレーション）

**なぜ:** UIは不安定である。クラッシュ、フリーズ、ユーザー操作ミスが起きる。P-1が要求する「絶対的な介入権」は、UIの障害から物理的に隔離されなければ保証できない。UIが死んでもCore Daemonは動き続け、エージェントを安全に制御できる状態を維持する。

**判定基準:** そのロジックはCore Daemon単体でCLIバイナリとして起動した場合に動作するか？ UIプロセスが存在しない環境でも全機能が利用可能か？ 動作しないなら、設計が間違っている。

### INV-2: Deterministic Boundary（決定論的境界）

> **S-IDEの振る舞いは、同じ入力に対して同じ出力を返さなければならない。**

**奉仕する理念:** P-1（決定論的オーケストレーション）

**なぜ:** P-1の「決定論的な環境」を実現するための直接的なルール。基盤が非決定的だと、障害の原因特定が不可能になり、エージェントの暴走を検出・停止できなくなる。確率的要素はS-IDEの境界の外（AIモデル側）に追い出す。

**判定基準:** そのAPIやモジュールは、モック入力に対してテストで検証可能な決定的出力を返すか？ 返さないなら、確率的要素を境界の外に追い出せ。

### INV-3: MCP as Universal Contract（MCPによる普遍的契約）

> **エージェント間通信、外部ツールアクセス、システムへの介入はすべてMCPを通じて行う。**

**奉仕する理念:** P-1（決定論的オーケストレーション）+ P-2（統合管理）

**なぜ:** P-1が要求する「記録可能・傍受可能・遮断可能な通信」と、P-2が要求する「体系的なエージェント管理」の両方を、単一の契約レイヤーで実現する手段。契約なき通信はN²の結合を生み、可観測性と管理可能性の両方を崩壊させる。

**判定基準:** そのエージェント間のデータ交換はMCPサーバーを経由しているか？ 直接importや共有グローバル変数を使っていないか？

### INV-4: Config-as-Code（宣言的管理）

> **環境依存の値はコードにハードコードしない。設定ファイルまたは環境変数で宣言的に管理する。**

**奉仕する理念:** P-2（統合管理）

**なぜ:** P-2が要求する「再現可能な構造」の基盤。ハードコードされた値は暗黙知となり、再現性を破壊する。エージェント構成を任意の環境にデプロイするには、すべての環境依存値が宣言的に管理されていなければならない。

**判定基準:** そのポート番号、パス、APIキー、閾値は設定解決チェーン（env → file → default）を通っているか？

### INV-5: Context Economy（コンテキスト経済性）

> **エージェントに提供する情報は、生データではなく抽象化された表面状態のみとする。**

**奉仕する理念:** P-2（統合管理）

**なぜ:** P-2が要求する「有限なコンテキストを考慮した最適な情報提供」の実装ルール。生ログ・生出力はコンテキストウィンドウを食い潰し、トークン消費が増え、推論精度が下がり、コストが増大する。エージェントの能力を最大限に引き出すには、情報を構造化・差分化して提供する必要がある。

**ただし:** 可観測性（INV-6, P-1）との衝突時は、INV-6が優先される（§2.1参照）。モニタリングに必要なデータは、効率を犠牲にしてでも保持せよ。

**判定基準:** そのデータ提供は、受け取るエージェントのコンテキスト効率を考慮しているか？ 生ストリームを流していないか？ ただし、INV-6に必要な観測データを削っていないか？

### INV-6: Observable & Interruptible（可観測性と介入権）

> **すべてのエージェント実行は、外部から観測可能で、いつでも遮断可能でなければならない。**

**奉仕する理念:** P-1（決定論的オーケストレーション）

**なぜ:** P-1の「絶対的な可観測性と介入権」を直接実現するルール。AIエージェントは予測不能な振る舞いをし得る。コスト超過、無限ループ、誤ったファイル操作。閾値を超えた場合に即座にプロセスを停止できる仕組みがなければ、ゼロトラストなタスク委譲は不可能である。

**判定基準:** そのエージェントの実行状態（トークン使用量、経過時間、操作回数）は外部からリアルタイムに取得可能か？ 強制停止APIは存在するか？

### Invariants対応表

| Invariant | 理念 | 優先度 | 役割 |
|---|---|---|---|
| INV-1: Headless-First | P-1 | 高 | 介入権のUI障害からの物理的隔離 |
| INV-2: Deterministic Boundary | P-1 | 高 | 決定論的環境の直接実現 |
| INV-3: MCP Universal Contract | P-1 + P-2 | 高 | 通信の記録・傍受・遮断 + 管理基盤 |
| INV-4: Config-as-Code | P-2 | 標準 | 再現可能な環境構成 |
| INV-5: Context Economy | P-2 | 標準 | 推論精度の保護（INV-6に劣後） |
| INV-6: Observable & Interruptible | P-1 | 高 | 絶対的介入権の実現 |

## 4. Evaluation Framework（評価指標）

エージェントは変更を加える前後に、以下の指標で品質を自己評価せよ。
各指標は §2 の理念と直結しており、スコアが下がる変更は理念への違反を示唆する。

### 4.1. Headless Independence Score（HIS）

> *その変更を加えた後、Core Daemon単体で全機能が動作するか？*
>
> **理念との連結:** P-1 — UIの障害から介入権を隔離できているか

| Score | 基準 |
|---|---|
| ◎ | 新しいロジックは100% Core Daemon側に実装されている |
| ○ | Core Daemon側がプライマリで、UIは表示のみ |
| △ | UIにビジネスロジックが一部漏れている |
| ✗ | UIがなければ動作しない機能を作った |

### 4.2. Context Efficiency Score（CES）

> *エージェントに提供する情報量は最小限か？*
>
> **理念との連結:** P-2 — エージェントの推論精度を保護しているか

| Score | 基準 |
|---|---|
| ◎ | 差分のみ提供、構造化データ |
| ○ | フィルタリングされたデータ |
| △ | 生データだが量が限定的 |
| ✗ | 生ログ・生ストリームの垂れ流し |

### 4.3. Contract Adherence Score（CAS）

> *すべての通信がMCP契約を通っているか？*
>
> **理念との連結:** P-1 + P-2 — 通信の可観測性とエージェント管理の一貫性

| Score | 基準 |
|---|---|
| ◎ | MCPサーバー経由、型安全、エラーハンドリング完備 |
| ○ | MCP経由だが型が一部不明確 |
| △ | 一部で直接依存がある |
| ✗ | 共有グローバル変数やイベントバスで通信 |

### 4.4. Determinism Score（DS）

> *その変更はテストで検証可能な決定的出力を持つか？*
>
> **理念との連結:** P-1 — S-IDEが決定論的環境として機能しているか

| Score | 基準 |
|---|---|
| ◎ | 入力に対して出力が一意に決まり、テストが存在する |
| ○ | 決定的だがテストが未実装 |
| △ | 時刻やランダム値への依存があるが注入可能 |
| ✗ | 非決定的で制御不能な外部状態に依存 |

### 4.5. Interruptibility Score（IS）

> *その処理は外部から観測・停止可能か？*
>
> **理念との連結:** P-1 — 絶対的介入権が維持されているか

| Score | 基準 |
|---|---|
| ◎ | ステータスAPI + 強制停止API + リソース上限が設定済み |
| ○ | ステータスは取得可能だが停止は手動 |
| △ | ログ出力のみで構造化された観測手段なし |
| ✗ | fire-and-forgetで観測も停止も不可能 |

## 5. Decision Framework（判断フレームワーク）

設計・実装判断に迷ったとき、以下の思考ステップを順に実行せよ。

```
Step 1: この変更はどちらの理念 (P-1/P-2) に奉仕するか？
         → どちらにも奉仕しないなら、その変更は本当に必要か再検討せよ。

Step 2: この変更は Invariants (§3) のどれかに違反するか？
         → 違反するなら、その設計は間違っている。やり直せ。
         → Invariant同士がコンフリクトするなら、§2.1 の優先順位に従え。

Step 3: Evaluation Framework (§4) の各スコアは維持または向上するか？
         → 下がるスコアがあるなら、トレードオフを明文化して判断を仰げ。

Step 4: この変更のスコープは正しいか？
         → ビジネスロジック・PTY管理・エージェント制御 → Core Daemon (Rust + axum/tokio)
         → UI表示・ユーザー入力の中継 → Web Client (React/TypeScript)
         → デスクトップ固有（ウィンドウ管理、Daemon起動） → Desktop Shell (Tauri/Rust)
         → スコープが複数にまたがるなら、分割できないか検討せよ。

Step 5: 影響範囲を特定したか？
         → レイヤー構成:
            Core Daemon  ← 全ビジネスロジックの唯一の所在地
            Web Client   ← Core DaemonのAPI/WebSocketを消費する薄いUI
            Desktop Shell← Core Daemonのプロセスライフサイクルを管理
         → 禁止: Web Client にビジネスロジックを置く
         → 禁止: Desktop Shell にビジネスロジックを置く
         → 禁止: UI層から Core Daemonの内部モジュールを直接import

Step 6: 変更前に読むべきspecドキュメントを特定したか？
         → §6 Document Map を参照せよ。
```

## 6. Document Map

本プロジェクトのドキュメントは以下の階層で構成される。
エージェントは**タスクに必要なドキュメントだけを読め**。全部読む必要はない。

### 6.1. 階層構造

```
ARCHITECTURE.md (本ドキュメント)     ← 全変更の前に必ず参照
    │
    ├── AGENTS.md                    ← 複数エージェント協調時に参照
    │     役割定義、命令系統、quality gate
    │
    ├── CLAUDE.md                    ← Claude Codeとして作業開始時に参照
    │     エントリポイント、必須コマンド
    │
    └── docs/specs/                  ← 担当ドメインのspecのみ参照
          ├── system.md              ← レイヤー構造・プロセスモデル・境界
          ├── server.md              ← Core Daemon API・ミドルウェア・DB・設定
          ├── frontend.md            ← コンポーネント・状態管理・API Client
          ├── desktop.md             ← Tauri/Rust Shell・Daemonライフサイクル
          ├── agents.md              ← エージェントシステム・MCP・通信
          ├── terminal-io.md         ← PTY管理・WebSocket・表層的I/O
          └── conventions.md         ← 命名規則・テスト・ビルド・依存関係
```

### 6.2. ルーティングテーブル（タスク種別 → 読むべきドキュメント）

| タスク | 必須 | 推奨 |
|---|---|---|
| 新しいAPIルートを追加 | `server.md` | `conventions.md` |
| UIコンポーネントを追加 | `frontend.md` | `conventions.md` |
| エージェント機能を追加 | `agents.md` | `terminal-io.md` |
| ターミナル機能を変更 | `terminal-io.md` | `server.md` |
| デスクトップ固有機能 | `desktop.md` | `system.md` |
| 型定義を変更 | `conventions.md` | 影響先のspec |
| アーキテクチャ判断 | `system.md` | 本ドキュメント再読 |
| テストを追加 | `conventions.md` | 対象のspec |
| 複数エージェント協調 | `AGENTS.md` | `agents.md` |
